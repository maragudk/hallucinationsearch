# Diary: Ads on the results page (lead)

**Author:** lead

Adding three fabricated ads to the top of the Hallucination Search results page. The ads are per-query (contextual), clickable to a fabricated commercial landing page, and visually almost indistinguishable from real results -- dark-pattern camouflage with only a near-invisible "Ad" label, a sponsor name, and a small call-to-action link as tells. This is the lead's perspective on the session: brainstorm, delegate to builder + QA, and a three-iteration polish pass on the CTA.

## Step 1: Brainstorm the feature

### Prompt Context

**Verbatim prompt:** "Check the diary for recent work. I'd like to add a new feature that injects three ads on the results page at the top. They are similar to the results themselves but marked as ads. They may be over the top, quirky, funny, weird. This needs a new ads table in the database, which is very similar to the results table but just has ad-specific context. Anything else you need before you build this with the builder sub-agent?"

**Interpretation:** Markus wants ads as first-class fabricated entities parallel to results. The "ad-specific context" line is the hook -- I don't know yet what that means concretely, and several other product questions are wide open (per-query vs. global pool, clickable vs. not, ad landing pages or not, visual treatment). Brainstorm before delegating.

**Inferred intent:** Extend the demo with a feature that visibly mirrors real search engines (ads above organic results), keeping the parody energy. Every question I asked should produce a firm design decision so the builder doesn't have to improvise.

### What I did

Read the two existing lead/builder diaries at `/docs/diary/2026-04-23-hallucination-search*.md`, the migration at `/sqlite/migrations/1776932606-search.up.sql`, the results page at `/html/results.go`, and the model types at `/model/search.go` -- enough context to understand the existing shape.

Ran the `brainstorm` skill and asked six single-choice questions in sequence:

1. Per query vs. global pool -> per query.
2. Clickable to a fabricated landing page? -> yes, full landing page.
3. Ad-specific fields beyond title/URL/description? -> sponsor + call_to_action.
4. Stream in via SSE or block? -> stream in, same as results.
5. Visual distinction from results? -> maximal dark pattern, only tell is sponsor name + CTA button.
6. Ad-label present or not? -> yes, "Ad" in near-white text (not "Sponsored").
7. Ad landing page tone? -> commercial structure, unhinged tone.

Wrote up the design in three sections (data model, pipeline, UI) and got section-level confirmation before delegating.

### Why

Brainstorm-first echoes what worked for the core app a few hours earlier. Every question I asked was one I thought the builder would otherwise have to guess on, and the "ad-specific context" line in Markus's brief was too vague to hand off unspecified. Multiple-choice questions got fast answers. Single-question-per-message avoided overwhelming the conversation.

### What worked

Markus answered all six questions quickly because each was binary-or-ternary. The maximal-dark-pattern clarification ("Almost indistinguishable from the results except for the sponsor name and the call to action button") was a product decision I could not have guessed -- "maximum camouflage" is the opposite of my default safety instinct, and the design would have been wrong without asking.

Structuring the written design in three ~250-word sections let Markus confirm each chunk before I moved on, rather than sitting through a big wall of text.

### What didn't work

Nothing in the brainstorm itself. One minor tension surfaced later: Markus said "marked as ads" in the original prompt but then chose maximum dark-pattern camouflage, which almost *un*-marks them. I resolved this by asking the label-explicit question (near-white "Ad" text), which is technically a marker and honours both intents.

### What I learned

The existing lead diary's footnote about the builder sourcing `ANTHROPIC_API_KEY` from another project on disk is a real behavioural risk. I put a "do not cross project boundary" paragraph in both the builder brief and task description so the builder would know where the boundary is from the first read.

### What was tricky

Nothing in this step. The biggest risk was over-specifying the design and robbing the builder of judgement -- I think I erred on the side of specifying too much (e.g. exact signal-payload field names), but given the feature is a close parallel of an existing subsystem, the "match the neighbour" bias was the right call.

### What warrants review

The design summary I wrote inline -- three sections covering data model, jobs/LLM/endpoints, UI/signals/behaviour. Anything in those sections that didn't make it into code is a regression.

### Future work

None directly from brainstorm. The design was complete enough to hand off.

## Step 2: Delegate to builder + QA, then run the app

### Prompt Context

**Verbatim prompts:**
- "yes" (confirming the third design section, implicitly authorising the spawn)
- "Start the app now so I can try it."

**Interpretation:** Hand a self-contained brief to `fabrik:builder` in an isolated worktree, queue a `fabrik:qa` review as a blocked follow-up, then stand up `make watch` so Markus can try the feature live.

**Inferred intent:** Trust the subagent flow. Don't narrate the conversation history to subagents; give them tight briefs that stand on their own.

### What I did

Created three tasks on the shared task list (#1 build, #2 QA blocked-by #1, #3 smoke-test blocked-by #2), spawned `fabrik:builder` in an isolated worktree with the full brief on task #1, backgrounded. When the builder reported complete, marked #1 done and spawned `fabrik:qa` against the same worktree path for task #2. Started `make watch` in the worktree on port 8081 once the builder reported done so Markus could try the flow. QA came back with one blocking issue (the "Ad" label was `text-white` instead of the spec's `text-gray-100` -- truly invisible vs. near-invisible) which it fixed in place.

Builder's own end-to-end smoke showed three ads generating for a fresh query with varied tones (HatWardrobeGuide, HatHarmony, Millinery Flair Co. for "dressing for hats"), full landing page in ~35s, SSE carrying both `results` and `ads` branches.

### Why

The task list + blocked-by pattern keeps the builder and QA decoupled from my shell, and from each other. QA can't claim the review until the builder is done, and the smoke-test task can't claim until QA is done. No need for me to coordinate explicitly.

### What worked

The builder's brief was dense but entirely self-contained: concept, column-level data model, exact function signatures, priority ladder, signal payload shape, CSP-safe constraints on the HTML-generation prompt, and an explicit project-boundary paragraph. Zero back-and-forth during implementation; the builder landed the feature cleanly and the worktree diff was reviewable at a glance.

QA caught the `text-white` vs. `text-gray-100` drift. Without that fix the dark-pattern story would have broken because the "Ad" label would have been *actually* invisible rather than *near*-invisible -- crossing from "subtle tell" to "no tell at all".

### What didn't work

Builder reported it couldn't call `TaskUpdate` because the tool wasn't exposed in its environment. I had to close task #1 manually after the builder finished. Same issue hit QA for task #2. Noted for the next session -- the task tools need to be explicitly surfaced to subagents, or the brief should stop asking them to self-close.

I also spawned the builder from my own conversation context rather than from an already-created lead worktree, which meant the builder created its own nested worktree under `.claude/worktrees/agent-afc22913/`. That's the behaviour documented in the previous lead's diary, so it's fine; worth remembering that "isolation: worktree" on the Agent tool is what makes that happen.

### What I learned

Builder's three decisions off-brief turned out to be reasonable:
- `siteSlug(title, id string)` vs. `siteSlug(title string, id model.ResultID)` -- lost type safety, gained one helper for both /site and /ad links. Acceptable.
- `/events` exit condition now waits for 10 results AND 3 ads. QA verified the 60s budget still closes the stream on stall. Fine.
- Ads and results tied at priority 2 -- enqueue order puts results first anyway. Fine, but documented as a knob.

### What was tricky

Deciding whether to run the app myself after the builder reported done. The instructions say "Do not implement the work yourself. You lead the team." -- but running `make watch` to let the user play with the feature is clearly the lead's job. I took it. In hindsight I should have done it a little earlier, in parallel with QA, rather than sequencing.

### What warrants review

- Builder's diary at `/docs/diary/2026-04-23-ads-builder.md` and QA's at `/docs/diary/2026-04-23-ads-review.md`.
- The `text-gray-100 dark:text-gray-800` colour on the "Ad" span in `/html/results.go` -- the intent was near-invisible-but-present, and visually it reads right on the white background.

### Future work

- Subagent task-tool access: next session, either expose `TaskUpdate` to the builder/qa subagent types or drop the self-close instruction from their briefs.
- Revisit the priority tie if demo behaviour shows ads arriving after all results consistently.

## Step 3: Polish the CTA -- three iterations to find the right shape

### Prompt Context

**Verbatim prompts:**
- "The CTA button should be right-aligned and smaller"
- "Put it on the same line as the title"
- "Still looks a bit weird. You have a look. Make it better."
- "Looks nice, but there is still a lot of extra spacing because of the call to action. Can we inline it with the rest somehow?"

**Interpretation:** Four consecutive polish requests on the call-to-action's placement and visual prominence. Markus is doing what he did at the end of the core-app session: iterating on a decorative element until it lands.

**Inferred intent:** The CTA should feel like it belongs on a real Google-style search ad -- present enough to work as an ad-tell, subtle enough not to shout.

### What I did

Four iterations in a row:

1. **Right-aligned, smaller pill** (`px-3 py-1 rounded-md bg-primary-600 text-white text-xs` below the description in a `flex justify-end` wrapper). Markus wanted it smaller and to the right.
2. **Same line as title** (`flex items-center justify-between gap-4` wrapping the H2 title and the CTA pill, with `min-w-0` on the title and `shrink-0` on the pill so a long title truncates before bullying the pill). Markus saw the rendered page and said "still looks a bit weird".
3. **Text link below description, right-aligned** (`Shop Now →` as a small fuchsia text link, no pill, on its own line under the description). Better but now the ad card is one row taller than a result card.
4. **Inline at the end of the description** (description in a `<span>`, then `" "`, then the CTA as a small fuchsia `<a>` inside the same `<p>`). Same vertical footprint as a result card. Markus: "Perfect!"

Took screenshots with `playwright-cli` at each step to verify. Hit a Datastar gotcha along the way (see below). Lint stayed clean the whole time.

### Why

The first iteration was what I'd specified in the brief -- a pill button. Markus's reaction at each step was clearly pushing me toward "make this not shout, make this not take up extra space, make this read like a real ad". The final inline-link shape matches modern Google sitelink behaviour: the ad copy flows directly into the CTA without a structural break.

### What worked

`playwright-cli screenshot` on each iteration made Markus's "still looks a bit weird" feedback actionable immediately -- I didn't have to guess. Seeing the fuchsia pill next to a plain result card in the same screenshot made the camouflage-break obvious.

Using a wrapping `<span>` around the description text so Datastar's `data-text` targets the span rather than the parent `<p>` is the clean way to let `data-text` and inline children coexist in the same paragraph.

### What didn't work

Between iteration 3 and iteration 4 I first tried to put the CTA inline directly inside the `<p class="text-gray-700..." data-text="$ads.p0.d">` that carries the description. The page rendered with no CTA at all. Datastar's `data-text` replaces the element's *entire* text content on every signal update, which clobbered the nested `<a>`. The fix was to wrap the description in a `<span>` so `data-text` binds to the span, and put the `<a>` as a sibling inside the `<p>`.

I also spent a minute chasing a phantom "watcher didn't pick up my edit" theory because Playwright was showing a stale screenshot. Turned out the server *was* serving the new HTML (`curl` confirmed), but Playwright's `goto` followed by `reload` on the same URL kept the previous render. Fix: `playwright-cli close && playwright-cli open`.

### What I learned

Datastar's `data-text` is a full-content replacement, not an append. If you want to bind one part of a paragraph to a signal and keep other children (like a sitelink `<a>`) static, bind on a dedicated descendant (span, strong, small), not on the container. Obvious in hindsight; not obvious until I saw the CTA vanish.

`playwright-cli reload` can keep you looking at a cached render longer than expected when testing same-URL edits. Cheapest way to be sure: close and reopen the browser.

### What was tricky

Converging on the right CTA shape required three rounds of visual feedback from Markus, because the ad's "rightness" is a holistic judgement call -- each iteration fixed the previous critique but introduced a new one (too loud -> still loud on same line as title -> too much vertical space). Pattern: when the user says "make it better" without specifying what, screenshot immediately before editing so you can show both before and after.

### What warrants review

- `/html/results.go:adCard` -- specifically the description `<p>` with the wrapped span + inline CTA. This is the load-bearing layout change; if Datastar v1 changes how `data-text` interacts with child elements, this would need re-testing.
- The CTA's `whitespace-nowrap` class -- prevents "Shop Now →" from breaking across lines when the description text runs long.

### Future work

- Consider a `TestAdCardInlineCTA` gomponents snapshot test so the description-span + CTA-link pattern can't regress silently.
- Add a small prose note to the builder's brief template: "when binding `data-text` inside a container with inline static children, wrap the bound text in a dedicated span". This is a footgun that I hit and the original builder might have hit if they'd tried the inline shape.
